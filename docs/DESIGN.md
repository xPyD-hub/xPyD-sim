# xPyD-sim Design Document

## Overview

xPyD-sim is an OpenAI-compatible LLM inference simulator for the xPyD ecosystem. It simulates prefill and decode behavior with configurable latency, enabling testing of xPyD-bench and xPyD-proxy without real GPU hardware.

## Architecture

### One Codebase, Three Modes

The simulator runs as a single server with three modes:

- **dual** — full prefill + decode pipeline (aggregated deployment)
- **prefill** — acts as a prefill node in PD-disaggregated deployment
- **decode** — acts as a decode node in PD-disaggregated deployment

Internally, the logic is unified. Mode only affects latency configuration:

| Phase | Dual | Prefill | Decode |
|---|---|---|---|
| Prefill compute | Normal | Normal | 0 (not applicable) |
| KV transfer | 0 (local) | Sync: included / Async: 0 | Normal (wait for KV) |
| Decode compute | Normal | 1 token only | Normal |

### How PD Disaggregation Works (with xPyD-proxy)

1. Proxy sends request to **prefill node** with `max_tokens=1`
2. Prefill node computes prefill → KV cache is ready → returns 1 token as HTTP response
3. Proxy receives prefill response → sends full request to **decode node**
4. Decode node waits for KV transfer → starts generating tokens
5. Proxy merges both streams for the client

The prefill node's 1-token response acts as a signal that KV is ready. The user's actual first token comes from the decode node.

**User-perceived TTFT** = prefill compute + KV transfer + decode first token

## Three Latency Parameters

1. **Prefill delay** — time to process input tokens, scales with batch size
2. **KV transfer delay** — time to transfer KV cache from prefill to decode, scales with KV size (≈ input tokens)
3. **Decode delay** — time per output token, scales with batch size

## Configuration

### Simple Mode (fixed values)

```yaml
mode: dual
prefill_delay_ms: 50        # fixed prefill latency
kv_transfer_delay_ms: 5     # fixed KV transfer latency
decode_delay_per_token_ms: 10  # fixed per-token decode latency
```

### Profile Mode (curve fitting)

Users provide measured data points at different batch sizes. The `calibrate` tool fits a curve and generates a profile file.

```yaml
mode: decode
profile: profile.yaml  # generated by xpyd-sim calibrate
```

Example calibration input:

```yaml
prefill:
  - batch_size: 128,   delay_ms: 5
  - batch_size: 1024,  delay_ms: 30
  - batch_size: 4096,  delay_ms: 150
kv_transfer:
  - batch_size: 128,   delay_ms: 1
  - batch_size: 1024,  delay_ms: 5
  - batch_size: 4096,  delay_ms: 20
decode:
  - batch_size: 1,   context_length: 512,   delay_per_token_ms: 7
  - batch_size: 1,   context_length: 2048,  delay_per_token_ms: 9
  - batch_size: 1,   context_length: 8192,  delay_per_token_ms: 14
  - batch_size: 16,  context_length: 512,   delay_per_token_ms: 9
  - batch_size: 16,  context_length: 2048,  delay_per_token_ms: 12
  - batch_size: 16,  context_length: 8192,  delay_per_token_ms: 18
  - batch_size: 64,  context_length: 512,   delay_per_token_ms: 14
  - batch_size: 64,  context_length: 2048,  delay_per_token_ms: 19

```

Minimum 3 points per curve, recommended 5 for better accuracy.

### Decode Delay: 1D vs 2D Fitting

Decode speed depends on two factors:
1. **Batch size** — more concurrent requests = slower per token
2. **Context length** — longer KV cache = slower attention

Decode delay is always fitted as a function of **(batch_size, context_length)** — both dimensions are required. Minimum 9 sample points (3×3 grid), recommended 16 (4×4).

```yaml
decode_2d:
  - batch_size: 1,   context_length: 512,   delay_per_token_ms: 7
  - batch_size: 1,   context_length: 2048,  delay_per_token_ms: 9
  - batch_size: 1,   context_length: 8192,  delay_per_token_ms: 14
  - batch_size: 16,  context_length: 512,   delay_per_token_ms: 9
  - batch_size: 16,  context_length: 2048,  delay_per_token_ms: 12
  - batch_size: 16,  context_length: 8192,  delay_per_token_ms: 18
  - batch_size: 64,  context_length: 512,   delay_per_token_ms: 14
  - batch_size: 64,  context_length: 2048,  delay_per_token_ms: 19
  - batch_size: 64,  context_length: 8192,  delay_per_token_ms: 30
```

The calibrate tool uses scipy griddata or RBF for 2D surface fitting.

During runtime, in 2D mode, context_length grows as tokens are generated (context = input_tokens + generated_so_far), so decode delay naturally increases over the generation — this is realistic.


### Calibrate Tool

```bash
xpyd-sim calibrate --input sample_points.yaml --output profile.yaml --plot profile.png
```

**Outputs:**
1. `profile.yaml` — curve parameters for sim to load at runtime
2. `profile.png` — visualization showing sample points (scatter) + fitted curves, for user review

## EOS Simulation

Real models produce variable-length outputs. The simulator mimics this:

- Random output length: `uniform(max_tokens * eos_min_ratio, max_tokens)`
- Default `eos_min_ratio: 0.5` (configurable)
- `finish_reason: "stop"` when EOS fires, `"length"` when hitting max_tokens
- `ignore_eos: true` → always output full max_tokens
- Works in both streaming and non-streaming modes

## OpenAI API Compliance

- Accept ALL OpenAI API parameters without errors
- Response JSON format must exactly match OpenAI spec
- Streaming: first chat chunk delta must include `role: "assistant"`
- Streaming: final chunk includes `usage` when `stream_options.include_usage` is set
- All responses include `system_fingerprint`
- `logprobs` field present in choices (null when not requested)
- Content is dummy/fake, but format is spec-compliant

## Warm-up Simulation

Real GPUs have cold-start overhead (CUDA kernel compilation, etc.). Configurable warm-up:

- `warmup_requests: 3` — first N requests get extra latency penalty
- `warmup_penalty_ms: 200` — additional delay during warm-up
- After warm-up, latency returns to normal

When `stop` parameter is provided, the simulator actually truncates output:
- During generation, check if output contains any stop sequence
- If found: truncate at that point, set `finish_reason: "stop"`
- Works in both streaming and non-streaming modes
- In streaming mode, stop checking happens per-chunk

## Scheduling & Batching (Critical)

The simulator must model realistic scheduling behavior — requests compete for resources, affecting each other's latency. This is the core value of xPyD-sim.

**Simplifications:**
- No prefix caching
- No chunked prefill
- No KV cache preemption

### Prefill Scheduling

Prefill is **blocking** — while a batch is being processed, new requests can only queue. Only after the current batch completes does the scheduler look at the queue to form the next batch.

**Constraints (configurable):**
- `max_model_len` — max input sequence length for a single request (reject if exceeded)
- `max_num_batched_tokens` — max total tokens in one prefill batch
- `max_num_seqs` — max number of requests in one prefill batch

**Batch formation (runs after current batch completes):**
```
current_batch = []
current_tokens = 0

for request in queue (FIFO):
    if request.input_len > max_model_len:
        reject request with error
        continue
    if current_tokens + request.input_len > max_num_batched_tokens:
        break
    if len(current_batch) >= max_num_seqs:
        break
    current_batch.append(request)
    current_tokens += request.input_len

process current_batch → delay = f(current_tokens)  # from 1D prefill profile
move all completed requests to decode queue
```

### Decode Scheduling

Decode also operates at **iteration granularity** — batch membership can only change between iterations, not during.

**Each decode iteration:**
1. Generate 1 token for every request in the current decode batch
2. Delay for this iteration = f(current_batch_size, avg_context_length) from 2D decode profile
3. Stream the generated token to each request's client
4. Check completions: remove requests that hit EOS or max_tokens (batch out)
5. Check waiting queue: add new requests that fit constraints (batch in)
6. Go to next iteration

**Constraints:**
- `max_num_seqs` — max concurrent requests in decode batch (same param as prefill, or separate)
- No KV memory simulation — we don't track GPU memory, just count sequences

**Key behavior:**
- When a new request joins, batch_size increases → everyone's next token is slower
- When a request leaves, batch_size decreases → everyone speeds up
- Context length grows each iteration → later tokens naturally slower than earlier ones

### Engine Loop (Revised)

```
while running:
    # Prefill phase (blocking)
    if prefill_queue is not empty and no prefill batch running:
        batch = form_prefill_batch(queue, max_num_batched_tokens, max_num_seqs)
        delay = prefill_profile.lookup(sum(r.input_len for r in batch))
        await sleep(delay)
        for r in batch:
            move r to decode_waiting_queue (with KV transfer delay if in decode mode)

    # Decode phase (one iteration)
    if decode_batch is not empty:
        # Batch in: add waiting requests
        for r in decode_waiting_queue:
            if len(decode_batch) < max_num_seqs:
                decode_batch.add(r)

        # Generate 1 token for all
        batch_size = len(decode_batch)
        avg_ctx = mean(r.context_length for r in decode_batch)
        delay = decode_profile.lookup(batch_size, avg_ctx)
        await sleep(delay)

        # Stream tokens + check completions
        for r in decode_batch:
            r.generated_tokens += 1
            r.context_length += 1
            stream_token(r)
            if r.is_done():  # EOS or max_tokens
                decode_batch.remove(r)
                finalize(r)
```
